## Production Order Rules (BR-O)

**BR-O01 — Sequential Status Transitions**
A production order can only move forward through its status lifecycle:
`DRAFT → PLANNED → IN_PRODUCTION → PRODUCTION_COMPLETE → CLOSED`
No backward transitions are permitted. Cancellation is not modelled (orders with DRAFT/PLANNED status may be deleted — **[UNKNOWN]**).

**BR-O02 — Parts Required Before Planning**
A production order cannot transition from DRAFT to PLANNED unless at least one `production_order_parts` record exists.

**BR-O03 — All Parts Released Before Starting**
A production order cannot transition from PLANNED to IN_PRODUCTION unless all `production_order_parts` records have status `RELEASED`.

**BR-O04 — All Stages Complete Before Production Complete**
A production order cannot transition from IN_PRODUCTION to PRODUCTION_COMPLETE unless all `production_stage_logs` for that order have status `COMPLETE`.

**BR-O05 — Packing Must Be Posted Before Closing**
A production order cannot transition to CLOSED unless its associated `packing_orders` record has status `POSTED`.

**BR-O06 — Document Number Auto-Generation**
`production_orders.order_number` must be generated by `DocumentNumberingService`. The service is global; do not re-inject the module. See GAP-016 in KEB-19 for known gap on document numbering configuration.

**BR-O07 — CMO Line Linkage**
If `release_type = FULL`, `cmo_line_id` is required (the entire CMO line is fulfilled by this order). If `release_type = PARTIAL`, `cmo_line_id` is nullable.

**BR-O08 — Stage Logs Initialized on Order Start**
When a production order moves to IN_PRODUCTION, `production_stage_logs` records for all stages on the assigned production line must be created with status `PENDING`.

## Material Release Rules (BR-R)

**BR-R01 — Release Only Reserved Bags**
Only `inventory_bags` records that are RESERVED for this specific production order may be included in a release group line. Releasing from a bag reserved for a different order or in any other status is forbidden.

**BR-R02 — Release Amount Constraint**
`dozens_to_release` per line must be ≤ the reserved dozens for that bag on this order. Over-release is rejected with `BadRequestException`.

**BR-R03 — Bag Status on Full Release**
When a bag's total `reserved_dozens` reaches zero after release, the bag status transitions to RELEASED. If partially released, bag status remains RESERVED.

**BR-R04 — CMO Line Decrement (Optimistic Lock)**
If the production order has a `cmo_line_id`, releasing material decrements `customer_manufacturing_order_lines.remaining_dozens`. This field uses optimistic locking (version). Conflict → retry up to 3 times, then `ConflictException`.

**BR-R05 — Atomic Release Group**
All lines within a release group must succeed atomically. Any line failure rolls back the entire group.

**BR-R06 — Release Group Sequence**
`release_groups.group_number` is auto-incremented per order (read max, increment). Concurrent release groups on the same order must use atomic increment or locking.

**BR-R07 — Part Status Update**
After any release group is created, if all bags for a given `part_id` in this order are now RELEASED, update `production_order_parts.status` → RELEASED for that part.

## Production Stage Rules (BR-S)

**BR-S01 — Sequential Stage Execution**
Stages are ordered by their position on the production line configuration. A stage cannot move to IN_PROGRESS until the previous stage is COMPLETE. The first stage can start as soon as the order is IN_PRODUCTION.

**BR-S02 — Conservation of Dozens**
For every stage:
```
output_dozens + scrap_dozens + incomplete_dozens = input_dozens
```
Violation is rejected with `BadRequestException`.

**BR-S03 — First Stage Input = Total Released**
The first stage's `input_dozens` equals the total dozens released for the order (sum of `release_group_lines.dozens_released`).

**BR-S04 — Subsequent Stage Input**
Each subsequent stage's `input_dozens` equals the `output_dozens` of the immediately preceding stage. (Incomplete dozens from a previous stage may be picked up as a supplementary — see BR-Sup04.)

**BR-S05 — Stage Completion Triggers WIP Update**
On a stage completing (status → COMPLETE), a `WIP_CONSUMPTION` inventory transaction must be recorded, and `wip_inventory` must be updated for each part covered by this order.

**BR-S06 — Last Stage Completion Triggers Quality Output**
When the final stage completes, the output is available for quality inspection. A `QUALITY_OUTPUT` recording must follow.

**BR-S07 — Scrap Records Are Optional Per Stage**
A stage completion with zero scrapped dozens requires no `scrap_records` rows. If `scrap_dozens > 0`, at least one `scrap_records` row is required.

**BR-S08 — Incomplete Records Are Optional Per Stage**
Same pattern as BR-S07: if `incomplete_dozens > 0`, at least one `incomplete_item_records` row is required.

## WIP Inventory Rules (BR-W)

**BR-W01 — Optimistic Locking Required**
Every write to `wip_inventory` must read the current `version`, include it in the update WHERE clause, and verify `rowCount = 1`. On conflict, retry up to 3 times with re-read, then raise `ConflictException`.

**BR-W02 — Non-Negative Balance**
`wip_inventory.dozens_in_wip` must never go below zero. Service layer enforces this before write.

**BR-W03 — Upsert Pattern**
On first stage completion for an `(order_id, part_id)` pair, create the `wip_inventory` record (initial version = 1). On subsequent updates, use the upsert/update-with-version pattern.

**BR-W04 — WIP Cleared on Return**
When material is returned to warehouse (BR-Rt), `wip_inventory.dozens_in_wip` is decremented for the returned part.

## Quality Output Rules (BR-Q)

**BR-Q01 — Optimistic Locking Required**
Every write to `quality_output_boxes` must use the version optimistic lock pattern (same as BR-W01).

**BR-Q02 — Accumulation Pattern**
Multiple quality output recordings for the same `(order_id, color_id, size_id)` accumulate (upsert-add). Each recording adds to `dozens_available`, not replaces it.

**BR-Q03 — Output Cannot Exceed Stage Output**
Total `dozens_passed` across all quality output recordings for an order must not exceed the final stage's `output_dozens`. **[UNKNOWN: whether this is enforced per-recording or only as a total]**

**BR-Q04 — QO Required Before Packing**
At least one `quality_output_boxes` record with `dozens_available > 0` must exist for an order before a packing order can be created.

## Packing Rules (BR-P)

**BR-P01 — Active Packing Pattern Required**
An active `packing_patterns` record must exist for the order's `model_id`. If no active pattern exists, packing order creation fails with `NotFoundException`.

**BR-P02 — Assembly Consumes Quality Output (Optimistic Lock)**
Each `dozen_assembly_lines` creation decrements `quality_output_boxes.dozens_available`. The optimistic lock version must be checked (BR-Q01 pattern).

**BR-P03 — Computed Columns Are Database-Generated**
`dozen_assembly_lines.dozens_consumed = pieces_consumed / 12` is DB-generated. Never write this column directly.
`packing_verifications.variance_dozens = system_count_dozens - physical_count_dozens` is DB-generated. Never write this column directly.

**BR-P04 — Verification Approval Is Mandatory**
A supervisor must call the approve endpoint even if `variance_dozens = 0`. This is a deliberate four-eyes control.

**BR-P05 — Post Requires Approved Verification**
`packing_orders.status` cannot move to POSTED unless `packing_verifications.is_approved = true`.

**BR-P06 — Post Creates Finished Goods Bags**
POSTing the packing order atomically creates `finished_goods_bags` records and `PACKING` inventory transactions.

**BR-P07 — One Packing Order Per Production Order**
`packing_orders.production_order_id` is unique. Attempting to create a second packing order for the same production order fails.

**BR-P08 — Packing Order Created on Production Complete**
A packing order can only be created after the production order reaches PRODUCTION_COMPLETE.

## Finished Goods Rules (BR-FG)

**BR-FG01 — FG Bags Created Only on Packing Post**
`finished_goods_bags` records are created exclusively by the `PostPackingOrderUseCase`. No standalone creation endpoint exists.

**BR-FG02 — Customer Assignment**
`finished_goods_bags.customer_id` comes from the CMO linked to the production order (via `cmo_line_id → cmo.customer_id`). For PARTIAL orders without CMO, **[UNKNOWN: customer assignment source]**.

**BR-FG03 — Warehouse Assignment**

## Return to Warehouse Rules (BR-Rt)

**BR-Rt01 — Can Only Return Released Material**
Return is only valid for material that was previously released to this order (a matching `release_group_line` with that `bag_id` must exist).

**BR-Rt02 — Return Amount Constraint**
`dozens_returned` ≤ dozens previously released for that bag in this order, minus any prior returns.

**BR-Rt03 — Inventory Transaction Required**
Every return creates a `RETURN` transaction in `inventory_transactions`.

**BR-Rt04 — Bag Status Revert**
If all dozens from a bag are returned, the bag status reverts to RECEIVED. If partially returned, bag remains RELEASED.

**BR-Rt05 — Part Status on Full Return**
If all WIP for a given `production_order_parts` entry reaches zero after return, the part status → RETURNED.

## Supplementary Material Rules (BR-Sup)

**BR-Sup01 — Negligence Requires Responsible Party**
If `reason = NEGLIGENCE`, at least one `supplementary_request_negligence` record naming the `responsible_user_id` is required. `GENUINE_SHORTAGE` requires no negligence records.

**BR-Sup02 — Request Number Auto-Generation**
`supplementary_material_requests.request_number` is auto-generated by `DocumentNumberingService`.

**BR-Sup03 — Sequential Status Transitions**
`DRAFT → PENDING_APPROVAL → APPROVED → TRANSFERRED`  
`PENDING_APPROVAL or APPROVED → REJECTED`  
`DRAFT or PENDING_APPROVAL or APPROVED → CANCELLED`  
TRANSFERRED and REJECTED are terminal states.

**BR-Sup04 — Production Order Must Be Active**
A supplementary request can only be created for an order with status IN_PRODUCTION. Cannot create for DRAFT, PLANNED, PRODUCTION_COMPLETE, or CLOSED orders.

**BR-Sup05 — Transfer Creates Inventory Transaction**
TRANSFERRED status triggers a `SUPPLEMENTARY_RELEASE` inventory transaction per request line.

**BR-Sup06 — Approval Authority**
Approval requires a role of MANAGER or above. **[UNKNOWN: whether a specific role check is required or general RBAC suffices]**.

## Inventory Integration Rules (BR-Inv)

**BR-Inv01 — Never Write Inventory Tables Directly**
The Production Module must not issue raw Prisma writes to `inventory_transactions` or `inventory_bags`. It must use the Inventory module's services or repositories via dependency injection.

**BR-Inv02 — Transaction Type Mapping**
Every production-triggered inventory movement uses the correct `TxnTypeEnum` value:

| Trigger | TxnType |
|---|---|
| Release group created | RELEASE |
| Stage completed | WIP_CONSUMPTION |
| Quality output recorded | QUALITY_OUTPUT |
| Packing order posted | PACKING |
| Material returned | RETURN |
| Supplementary transferred | SUPPLEMENTARY_RELEASE |

**BR-Inv03 — Atomic Inventory Movement**
Every inventory transaction (bag balance update + `inventory_transactions` insert) must be atomic (Prisma `$transaction`).

**BR-Inv04 — Event Emission After Inventory Write**
After any inventory transaction in the production flow, the corresponding event must be emitted via `ProductionEventPublisher`. Events are fire-and-forget; failures in event handlers must not roll back the inventory write.
