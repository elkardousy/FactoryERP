## 1. Aggregate Overview

```
CMO Line
  └── Production Order
        ├── Production Order Parts (which bags/parts are covered)
        ├── Release Groups
        │     └── Release Group Lines (which bags, how many dozens)
        ├── Production Stage Logs (per stage: input → output → scrap → incomplete)
        │     ├── Scrap Records
        │     └── Incomplete Item Records
        ├── WIP Inventory (live balance per order+part)
        ├── Quality Output Boxes (finished dozens per order+color+size)
        ├── Return Transactions (material returned to warehouse)
        └── Packing Order
              ├── Dozen Assemblies (packing pattern execution)
              │     └── Dozen Assembly Lines (pieces consumed per part)
              ├── Packing Verification (physical vs. system count)
              └── Finished Goods Bags (output bags sent to FG warehouse)

Production Order (standalone path)
  └── Supplementary Material Requests (extra material needed mid-production)
        └── Supplementary Request Lines
              └── Supplementary Request Negligence (if reason = NEGLIGENCE)
```

## 2. Aggregate Detail

### 2.1 Production Order (`production_orders`)

**Responsibility:** Root aggregate. Tracks what is being manufactured, for which CMO line, by which production line, and its current lifecycle status.

**Schema key fields:**
- `order_id` — BigInt PK
- `order_number` — unique string, auto-generated by `DocumentNumberingService`
- `status` — `OrderStatusEnum`: DRAFT → PLANNED → IN_PRODUCTION → PRODUCTION_COMPLETE → CLOSED
- `model_id` — FK to `garment_models`
- `line_id` — FK to `production_lines` (production-setup module)
- `release_type` — `ReleaseTypeEnum`: FULL | PARTIAL
- `cmo_line_id` — FK to `customer_manufacturing_order_lines` (nullable on PARTIAL)

**Lifecycle invariants:**
- Cannot move to IN_PRODUCTION unless all parts are RELEASED
- Cannot move to PRODUCTION_COMPLETE unless all stage logs are COMPLETE
- Cannot CLOSE unless Packing Order is POSTED
- Status transitions are one-directional (no backwards transitions)

**Status transition diagram:**
```
DRAFT → PLANNED → IN_PRODUCTION → PRODUCTION_COMPLETE → CLOSED
```

---

### 2.2 Production Order Parts (`production_order_parts`)

**Responsibility:** Lists which fabric parts (by `part_id`) are covered by this production order and their release status.

**Schema key fields:**
- `order_part_id` — BigInt PK
- `order_id` — FK to `production_orders`
- `part_id` — FK to `garment_parts`
- `status` — `PartStatusEnum`: PENDING → RELEASED → COMPLETE → RETURNED
- `@@unique([order_id, part_id])`

**Invariants:**
- Each part appears at most once per production order
- `RELEASED` means at least one `release_group_line` exists for this part
- `RETURNED` only when all WIP for this part is returned to warehouse

---

### 2.3 Release Groups (`release_groups`)

**Responsibility:** Groups a batch of material releases associated with a single production order. Allows partial releases in multiple waves.

**Schema key fields:**
- `release_group_id` — BigInt PK
- `order_id` — FK to `production_orders`
- `group_number` — sequence within order (1, 2, 3...)
- `@@unique([order_id, group_number])`

---

### 2.4 Release Group Lines (`release_group_lines`)

**Responsibility:** Each line represents one physical bag's contribution to a release group: how many dozens were released from which source warehouse.

**Schema key fields:**
- `release_line_id` — BigInt PK
- `release_group_id` — FK to `release_groups`
- `bag_id` — FK to `inventory_bags`
- `dozens_released` — numeric
- `source_warehouse_id` — FK to `warehouses`

**Invariants:**
- `dozens_released` must be ≤ available dozens in that bag at time of release
- Releasing a bag reduces `inventory_bags.dozens_available` and creates a `RELEASE` transaction in `inventory_transactions`
- Physical bag status transitions: RESERVED → RELEASED (when fully consumed from that bag)

---

### 2.5 Production Stage Logs (`production_stage_logs`)

**Responsibility:** Records the input, output, scrap, and incomplete dozens for each manufacturing stage within a production order.

**Schema key fields:**
- `log_id` — BigInt PK
- `order_id` — FK to `production_orders`
- `stage_id` — FK to `production_stages` (production-setup module)
- `status` — `StageStatusEnum`: PENDING → IN_PROGRESS → COMPLETE
- `input_dozens` — dozens entering this stage
- `output_dozens` — dozens exiting this stage (output ≤ input)
- `scrap_dozens` — dozens scrapped (computed: input - output - incomplete)
- `incomplete_dozens` — dozens not finished yet
- `@@unique([order_id, stage_id])`

**Invariants:**
- `output_dozens + scrap_dozens + incomplete_dozens = input_dozens` (conservation law)
- Stages are sequential per production line configuration
- A stage cannot begin until the preceding stage is COMPLETE
- Stage COMPLETE triggers WIP update and quality output recording

**Lookup note:** `@@unique([order_id, stage_id])` exists, but for queries filtering only by `order_id`, use `findMany()`. For lookup by `(order_id, stage_id)`, use `findFirst()` not `findUnique()`.

---

### 2.6 Scrap Records (`scrap_records`)

**Responsibility:** Detailed record of each scrap event within a stage, capturing type, quantity, and optionally color/size.

**Schema key fields:**
- `scrap_id` — BigInt PK
- `log_id` — FK to `production_stage_logs`
- `scrap_type` — `ScrapTypeEnum`: SEWING_DEFECT, MATERIAL_DEFECT, STAIN, DAMAGE, SIZE_ISSUE, OTHER
- `dozens_scrapped` — numeric
- `color_id` — FK to `colors` (nullable)
- `size_id` — FK to `sizes` (nullable)

---

### 2.7 Incomplete Item Records (`incomplete_item_records`)

**Responsibility:** Records why some dozens did not complete a stage, enabling tracking of WIP items that need rework or supplementary materials.

**Schema key fields:**
- `incomplete_id` — BigInt PK
- `log_id` — FK to `production_stage_logs`
- `reason` — `IncompleteReasonEnum`: MISSING_PART, MISSING_MATERIAL, PRODUCTION_ISSUE, OTHER
- `dozens_incomplete` — numeric
- `notes` — text (nullable)

---

### 2.8 WIP Inventory (`wip_inventory`)

**Responsibility:** Live balance of work-in-progress garments per production order and part. Updated at each stage completion.

**Schema key fields:**
- `wip_id` — BigInt PK
- `order_id` — FK to `production_orders`
- `part_id` — FK to `garment_parts`
- `dozens_in_wip` — numeric (current live WIP balance)
- `version` — BigInt (optimistic lock)
- `@@unique([order_id, part_id])`

**Invariants:**
- Updated on `WIP_CONSUMPTION` inventory transactions
- `version` must be read, checked, and incremented on every write — conflicts throw `ConflictException`
- `dozens_in_wip >= 0` always

---

### 2.9 Quality Output Boxes (`quality_output_boxes`)

**Responsibility:** Tracks how many finished dozens are available per color/size combination after passing quality checks.

**Schema key fields:**
- `box_id` — BigInt PK
- `order_id` — FK to `production_orders`
- `color_id` — FK to `colors`
- `size_id` — FK to `sizes`
- `dozens_available` — numeric (decremented as packing consumes)
- `version` — BigInt (optimistic lock)
- `@@unique([order_id, color_id, size_id])`

**Invariants:**
- Created on `QUALITY_OUTPUT` recording; upserted on subsequent quality output for same combination
- `version` required for optimistic lock on packing consumption
- `dozens_available >= 0` always

---

### 2.10 Return Transactions (`return_transactions`)

**Responsibility:** Records material returned from production back to a warehouse (unreleased or excess material).

**Schema key fields:**
- `return_id` — BigInt PK
- `order_id` — FK to `production_orders`
- `part_id` — FK to `garment_parts`
- `bag_id` — FK to `inventory_bags`
- `dozens_returned` — numeric
- `destination_warehouse_id` — FK to `warehouses`
- `return_reason` — text

**Invariants:**
- Creates a `RETURN` transaction in `inventory_transactions`
- Bag status reverts: RELEASED → RECEIVED (if fully returned)
- `production_order_parts.status` → RETURNED when all WIP for that part is zero and return is processed

---

### 2.11 Packing Patterns (`packing_patterns` + `packing_pattern_lines`)

**Responsibility:** Defines how pieces are assembled into dozens — the ratio of pieces per size/color needed to make one assembled dozen. Used during packing execution.

**Schema key fields (`packing_patterns`):**
- `pattern_id` — BigInt PK
- `model_id` — FK to `garment_models`
- `is_active` — boolean

**Schema key fields (`packing_pattern_lines`):**
- `pattern_line_id` — BigInt PK
- `pattern_id` — FK to `packing_patterns`
- `part_id` — FK to `garment_parts`
- `color_id` — FK to `colors`
- `size_id` — FK to `sizes`
- `pieces_per_dozen` — integer

**Invariants:**
- Only one active pattern per model at a time (`@@unique` on `[model_id, is_active]` when `is_active = true` — UNKNOWN, verify in schema)
- Pattern lines define the assembly formula for the packing stage

---

### 2.12 Packing Orders (`packing_orders`)

**Responsibility:** Root aggregate for the packing execution phase. One per production order.

**Schema key fields:**
- `packing_order_id` — BigInt PK
- `production_order_id` — FK to `production_orders` (unique — one packing order per production order)
- `status` — `PackingOrderStatusEnum`: DRAFT → IN_PROCESS → ASSEMBLED → VERIFIED → POSTED → CANCELLED

**Status transition:**
```
DRAFT → IN_PROCESS → ASSEMBLED → VERIFIED → POSTED
                                           ↘ CANCELLED
```

**Invariants:**
- Created when production order reaches PRODUCTION_COMPLETE
- POSTED triggers creation of `finished_goods_bags`
- CANCELLED only before POSTED

---

### 2.13 Dozen Assemblies (`dozen_assemblies`)

**Responsibility:** Records each assembly event in the packing order — a sequence of dozens assembled according to the packing pattern.

**Schema key fields:**
- `assembly_id` — BigInt PK
- `packing_order_id` — FK to `packing_orders`
- `assembly_sequence` — integer (order within packing)
- `dozens_assembled` — numeric
- `@@unique([packing_order_id, assembly_sequence])`

---

### 2.14 Dozen Assembly Lines (`dozen_assembly_lines`)

**Responsibility:** Breakdown of pieces consumed per part/color/size in one assembly event.

**Schema key fields:**
- `assembly_line_id` — BigInt PK
- `assembly_id` — FK to `dozen_assemblies`
- `quality_output_box_id` — FK to `quality_output_boxes`
- `pieces_consumed` — integer
- `dozens_consumed` — dbgenerated (`pieces_consumed / 12`)

**Invariants:**
- `dozens_consumed` is a database-generated computed column — do not write it directly
- Consuming from a `quality_output_box` requires optimistic lock check on `quality_output_boxes.version`

---

### 2.15 Packing Verifications (`packing_verifications`)

**Responsibility:** Physical count verification step before posting the packing order.

**Schema key fields:**
- `verification_id` — BigInt PK
- `packing_order_id` — FK to `packing_orders` (unique — one verification per packing order)
- `physical_count_dozens` — numeric (actual physical count by supervisor)
- `system_count_dozens` — numeric (what the system calculated)
- `variance_dozens` — dbgenerated (`system_count_dozens - physical_count_dozens`)
- `verified_by` — FK to `users`
- `is_approved` — boolean

**Invariants:**
- `variance_dozens` is database-generated — do not write it directly
- Packing order cannot move to VERIFIED until `is_approved = true`
- Supervisor must explicitly approve despite variance (zero or non-zero)

---

### 2.16 Finished Goods Bags (`finished_goods_bags`)

**Responsibility:** Represents packed goods bags stored in the finished goods warehouse, ready for shipping.

**Schema key fields:**
- `fg_bag_id` — BigInt PK
- `packing_order_id` — FK to `packing_orders`
- `model_id` — FK to `garment_models`
- `customer_id` — FK to `customers`
- `warehouse_id` — FK to `warehouses` (finished goods warehouse)
- `cmo_line_id` — FK to `customer_manufacturing_order_lines` (nullable)
- `dozens_packed` — numeric
- `color_id` — FK to `colors`
- `size_id` — FK to `sizes`

**Invariants:**
- Created on Packing Order POSTED
- Creates a `PACKING` transaction in `inventory_transactions`
- `dozens_packed` must equal total from assembly lines for that color/size

---

### 2.17 Supplementary Material Requests (`supplementary_material_requests`)

**Responsibility:** Formal request for additional material when the original release is insufficient, with approval workflow.

**Schema key fields:**
- `request_id` — BigInt PK
- `request_number` — unique string, auto-generated by `DocumentNumberingService`
- `order_id` — FK to `production_orders`
- `status` — `SupplementaryStatusEnum`: DRAFT → PENDING_APPROVAL → APPROVED → REJECTED → TRANSFERRED → CANCELLED
- `reason` — `SupplementaryReasonEnum`: NEGLIGENCE | GENUINE_SHORTAGE
- `requested_by` — FK to `users`

**Schema key fields (`supplementary_request_lines`):**
- `request_line_id` — BigInt PK
- `request_id` — FK to `supplementary_material_requests`
- `part_id` — FK to `garment_parts`
- `bag_id` — FK to `inventory_bags`
- `dozens_requested` — numeric

**Schema key fields (`supplementary_request_negligence`):**
- `negligence_id` — BigInt PK
- `request_id` — FK to `supplementary_material_requests`
- `responsible_user_id` — FK to `users`
- `notes` — text

**Invariants:**
- NEGLIGENCE requests must have at least one `supplementary_request_negligence` record naming the responsible party
- TRANSFERRED creates a `SUPPLEMENTARY_RELEASE` transaction
- CANCELLED only before PENDING_APPROVAL or APPROVED (not after TRANSFERRED)

---

## 3. Cross-Aggregate Relationships

```
inventory_bags ──────── release_group_lines (bag releases)
inventory_bags ──────── return_transactions (bag returns)
inventory_bags ──────── supplementary_request_lines (supplementary releases)
inventory_transactions ─ (all production inventory movements write here)
wip_inventory ─────────  (live WIP balance, optimistic lock)
quality_output_boxes ──  (finished dozens, optimistic lock)
customer_manufacturing_order_lines ── production_orders (via cmo_line_id)
```
