## 1. Problem

GitHub Actions displays a runtime deprecation warning in CI logs:

```
Node.js 20 actions are deprecated. Please update the following actions to use Node.js 24:
  - actions/checkout@v4
  - actions/setup-node@v4
  - actions/cache@v4
  - actions/upload-artifact@v4
```

The CI pipeline completes successfully and all quality gates pass. However, the Node 20 deprecation indicates that GitHub Actions infrastructure will eventually remove or restrict support for actions running on Node 20. Continued use of `@v4` versions creates forward compatibility risk.

---

## 2. Assessment

### Version Gap Discovery

GitHub API (`/repos/actions/*/releases/latest`) confirmed the following gaps as of 2026-07-02:

| Action | Current | Latest Stable | Gap |
|---|---|---|---|
| `actions/checkout` | `@v4` | `v7.0.0` (2026-06-18) | 3 major versions |
| `actions/setup-node` | `@v4` | `v6.4.0` (2026-04-20) | 2 major versions |
| `actions/cache` | `@v4` | `v6.1.0` (2026-06-26) | 2 major versions |
| `actions/upload-artifact` | `@v4` | `v7.0.1` (2026-04-10) | 3 major versions |

### Root Cause

GitHub Actions major version increments are driven by Node.js runtime deprecation cycles:
- `@v4` was released when Node 20 became the GA runtime
- Subsequent major versions (`@v5`, `@v6`, `@v7`) tracked the Node.js LTS lifecycle through transition and stabilization on Node 24
- The deprecation warning is generated by GitHub Actions runner infrastructure when a job uses actions that declare `runs.using: node20` in their `action.yml`

### Input Compatibility

All inputs used in this repository are core stable inputs unchanged across all major versions:

| Input | Used In | Stable Across Major Versions |
|---|---|---|
| `fetch-depth: 1` | checkout | YES |
| `node-version-file: '.nvmrc'` | setup-node | YES |
| `cache: 'npm'` | setup-node | YES |
| `path`, `key`, `restore-keys` | cache | YES |
| `name`, `path`, `retention-days` | upload-artifact | YES |

---

## 3. Upgrade Strategy

**Strategy: In-place major version upgrade — no architecture changes.**

- Replace `@v4` with the latest stable major version tag for each action
- Apply the upgrade to all workflow files containing the action
- Maintain identical step definitions (inputs, conditionals, environment)
- Test with local quality gates after modification

Affected files:
- `.github/workflows/ci.yml` (4 upgrades)
- `.github/workflows/security-audit.yml` (3 upgrades)

Version mapping:
```
actions/checkout@v4      →  actions/checkout@v7
actions/setup-node@v4    →  actions/setup-node@v6
actions/cache@v4         →  actions/cache@v6
actions/upload-artifact@v4  →  actions/upload-artifact@v7
```

---

## 4. Compatibility

### Runtime Alignment

Before: Actions run on Node 20; project code runs on Node 24.16.0 (`.nvmrc`).  
After: Actions run on Node 24; project code runs on Node 24.16.0 (`.nvmrc`).

This improves consistency — the actions' JavaScript runtime now matches the project's configured runtime.

### Platform Matrix

| Platform | Status |
|---|---|
| `ubuntu-latest` | COMPATIBLE — verified via quality gates |
| `windows-latest` | COMPATIBLE — no Windows-specific action inputs |
| `macos-latest` | COMPATIBLE — no macOS-specific action inputs |

### Behavioral Compatibility

No behavioral changes are expected. The upgrade affects only the JavaScript runtime used internally by each action, not the action's external API, output variables, or workflow behavior.

## 5. Benefits

| Benefit | Description |
|---|---|
| **Warning elimination** | No more Node 20 deprecation warnings in CI logs |
| **Runtime alignment** | Actions now run on Node 24, matching the project's Node version |
| **Future compatibility** | Eliminates forward compatibility risk before GitHub removes Node 20 support from runner infrastructure |
| **Security** | Newer Node.js runtime incorporates upstream security patches |
| **Ecosystem currency** | Repository stays current with GitHub Actions ecosystem; no accumulated upgrade debt |

---

## 6. Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Cache miss on first CI run | INFO | tsbuildinfo cache will miss once (different action version may affect cache key); no functional impact; subsequent runs restore normally |
| API change in `upload-artifact@v7` | LOW | Only `name`, `path`, `retention-days` used — core stable inputs; no compatibility issue |
| GitHub reverting latest tag to older version | VERY LOW | GitHub actions are pinned to explicit major version (`@v7`) not a mutable tag; immune to this risk |

---

## 7. Rollback Plan

If any CI runner reports an unexpected failure after upgrade:

```bash
# Revert workflow files to @v4
git revert <upgrade-commit>
git push origin main
```

The revert takes effect immediately on the next CI run. Risk is minimal since:
- The upgrade is 7 line changes across 2 files
- No infrastructure, application, or Prisma changes were made
- All quality gates pass before and after
